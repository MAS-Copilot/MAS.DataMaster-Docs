MAS.DynamicViews.Variable

变量系统是 MAS.DynamicViews 的数据核心

整个动态视图运行过程中：

  • 控件绑定
  • 通讯同步
  • 运行时刷新
  • 状态显示
  • 事件交互

最终都依赖变量系统完成数据驱动

变量系统目标

变量系统主要目标：

  • 统一管理变量定义
  • 支持结构体变量
  • 支持字段级绑定
  • 支持变量监控
  • 支持批量更新
  • 支持通讯映射
  • 支持运行时绑定
  • 支持引用追踪
  • 支持高频实时刷新

变量模型

当前变量系统采用结构体变量 + 字段模型架构

整体结构：

DynamicViewVariableModel
├── VariableId
├── Name
├── GroupName
├── AccessMode
├── IsEnabled
├── Description
├── Usage
├── Fields
└── Properties

其中：

属性 说明
VariableId 结构体变量唯一标识
Name 结构体名称
GroupName 结构体分组
AccessMode 结构体访问模式
IsEnabled 是否启用
Description 结构体描述
Usage 变量用途，例如普通结构体或配方结构体
Fields 字段定义集合
Properties 扩展属性

普通结构体

普通结构体用于描述设备状态、工艺参数、按钮命令等常规数据

新增普通结构体时，系统会自动创建一个默认布尔字段，便于快速配置和测试。实施时建议按设备或工艺对象组织字段，例如

Conveyor01
├── Running
├── Fault
├── SpeedSet
└── CurrentSpeed

配方结构体

配方结构体用于通过 Modbus 保持寄存器与配方任务交互

创建配方结构体时会生成：

  • 配方编号字段：用于表示配方唯一标识
  • 操作字段：0 表示无操作，1 表示存储配方，2 表示获取配方
  • 结果字段：表示任务执行结果
  • 耗时、错误码和数据长度字段
  • 配方数据区字段

配方数据长度实际有效范围为 100 到 1000。配方任务当前只支持 Modbus TCP 保持寄存器

为什么使用结构体变量

传统动态组态通常采用全局变量列表模式

例如：

Motor1Running
Motor1Speed
Motor1Alarm

这种方式存在问题：

  • 变量数量巨大
  • 难以组织
  • 难以维护
  • 不符合 PLC 数据结构
  • 通讯映射复杂
  • 引用关系混乱

因此当前系统采用结构体变量组织数据

例如：

Motor1
├── Running
├── Speed
├── Alarm
└── Temperature

这样更符合

  • PLC 数据块
  • 设备结构
  • 工业对象模型
  • 通讯映射方式

字段模型

字段模型用于描述结构体中的具体数据点

典型结构：

DynamicViewVariableFieldModel
├── FieldId
├── Name
├── VariableType
├── DefaultValue
├── FixedString
├── Description
└── Properties

其中：

属性 说明
FieldId 字段唯一标识
Name 字段名称
VariableType 字段数据类型
DefaultValue 字段默认值，运行时按 VariableType 转换
Description 字段描述
FixedString 固定字符串配置，仅字符串类型使用
Properties 扩展属性

字段类型切换时，系统会按类型重置默认值：

类型 默认值
Bool false
Int16 / UInt16 / Int32 / UInt32 0
Float / Double 0.0
String N/A

字符串字段需要配置固定长度字符串参数。固定长度默认 50 字节，默认编码为 ASCII，默认填充字符为 \0，并会去除末尾填充

访问模式

访问模式定义在结构体变量层。当前结构体变量整体共享访问模式，而不是每个字段单独配置访问模式

访问模式主要用于：

  • 变量选择过滤
  • 控件绑定限制
  • 通讯读写限制
  • 写回限制
  • 后续权限控制

常见模式：

  • ReadOnly：可被通讯轮询读取，可作为显示和状态绑定来源，不允许写回
  • WriteOnly：不参与通讯轮询读取，只允许绑定到可写通讯通道并写出
  • ReadWrite：既可读取也可写入，适合参数读写和双向交互

注意：访问模式配置在结构体层级，一个结构体下的字段共享同一访问模式

数据类型

字段通过VariableType描述数据类型，当前常见类型包括：

  • Bool
  • Int16
  • UInt16
  • Int32
  • UInt32
  • Float
  • Double
  • String

字段类型主要用于：

  • 默认值转换
  • 变量选择过滤
  • 控件绑定适配
  • 通讯数据解析
  • 通讯数据写回
  • 运行时值转换

变量筛选

变量选择场景中，通常需要支持筛选

常见筛选条件包括：

  • 分组
  • 数据类型
  • 访问模式
  • 启用状态
  • 关键字

筛选主要用于：

  • 控件绑定
  • 通讯通道绑定
  • 变量监控
  • 事件配置
  • 脚本变量选择

命名与校验

结构体名称和字段名称必须符合变量命名规则：

  • 不能为空
  • 必须以字母或下划线开头
  • 后续字符可以包含字母、数字或下划线
  • 同一工程内结构体名称不能重复
  • 同一结构体内字段名称不能重复
  • 结构体至少需要一个字段

删除结构体或字段前，系统会检查引用。如果仍被控件绑定、控件事件或通讯通道使用，需要先解除引用

批量更新

变量系统需要支持批量更新

原因是 PLC 通讯通常不是逐字段读取，而是按设备、通道或结构体批量读取

批量更新可以减少：

  • 通知次数
  • UI 刷新次数
  • Dispatcher 压力
  • 高频轮询开销

推荐运行时更新链路：

CommunicationService
    ↓
VariableService
    ↓
RuntimeBindingManager
    ↓
WPF Control

变量监控

变量监控用于观察运行时变量状态

通常显示：

  • 结构体名称
  • 字段名称
  • 数据类型
  • 当前值
  • 默认值
  • 访问模式
  • 启用状态
  • 更新时间

变量监控应读取运行时变量服务中的当前值，而不是直接依赖字段定义模型

运行时变量值规则：

  • 如果字段尚未收到通讯值或本地写入值，读取结果为字段默认值
  • 通讯轮询更新结构体时，会批量提交发生变化的字段
  • 只有值发生变化时才触发绑定刷新
  • 文本框和选中状态控件写入失败时，会恢复为变量中心中的当前值

变量操作

控件事件中的变量操作支持：

操作 Bool 数值类型 String
设置为 0 或 False false 0 空字符串
设置为 1 或 True true 1 "1"
取反 true/false 取反 0 与 1 切换 保持当前值
递增 保持当前值 加 1 保持当前值
递减 保持当前值 减 1 保持当前值

Int16、UInt16、Int32、UInt32 的递增递减会限制在类型范围内，避免溢出

引用追踪

变量系统需要支持引用追踪，引用追踪用于分析：变量字段被哪里使用

常见引用来源：

  • 控件绑定
  • 通讯通道
  • 事件配置
  • 后续报警配置
  • 后续趋势配置

引用追踪主要用于：

  • 删除检查
  • 重命名影响分析
  • 工程维护
  • 调试排查
  • 无效引用检测

克隆规则

变量模型和字段模型支持克隆

当前克隆方式基于 JSON 序列化和反序列化

克隆主要用于：

  • 编辑副本
  • 弹窗编辑
  • 撤销重做
  • 临时修改
  • 深拷贝模型

注意：克隆结果会复制当前对象数据，但应避免把运行时缓存、服务实例或临时状态放入变量定义模型中
