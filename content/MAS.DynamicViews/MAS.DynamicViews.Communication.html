MAS.DynamicViews.Communication

通讯系统负责：

  • 设备连接
  • 协议通讯
  • 变量读取
  • 安全写入
  • 多设备轮询
  • 通讯状态管理
  • 运行时生命周期管理

整个系统采用运行时调度 + 通讯服务 + 安全写入 + 状态反馈的分层架构。通信配置主要通过设备中心完成，变量定义主要通过数据中心完成

配置方法

典型配置流程：

  1. 在数据中心创建结构体变量，并按设备数据块顺序配置字段
  2. 设置结构体访问模式：只读、只写或读写
  3. 在设备中心添加 Modbus TCP 设备
  4. 配置设备 IP、端口、UnitId、超时、重试、字节序和字序
  5. 为设备添加通道，选择地址区和起始地址
  6. 将通道绑定到结构体变量
  7. 执行检查，确认设备名、通道名、地址和访问模式合法
  8. 同步工程数据到内存

未同步的设备中心改动不会参与运行时通讯

通讯目标

通讯系统主要目标：

  • 支持多设备并行通讯
  • 支持多协议扩展
  • 支持结构体读写
  • 支持实时变量同步
  • 支持安全写入
  • 支持设备状态监控
  • 支持运行时动态启动停止
  • 支持异常隔离
  • 支持断线恢复
  • 支持工业级一致性

设计原则

原则 说明
读写分离 读取与写入完全独立
安全提交 写入成功后才允许更新变量中心
多设备独立 单个设备异常不影响其它设备
状态弱依赖 状态服务仅用于显示，不参与安全控制
变量中心统一 所有运行时值统一进入变量中心
协议解耦 DynamicViews 不直接依赖具体协议实现
生命周期管理 统一由 RuntimeService 控制

职责分层：

层 主要职责
运行时编排 启停、连接设备、启动脚本和轮询、统一清理
轮询调度 选择可读上下文、按设备并发、设备内串行、按 Usage 分派
通信服务 把结构体读写转换为具体协议调用
协议访问 创建、缓存和配置失效后重建协议实例
变量存储 保存运行时值、比较变化、发布 ValueChanged
UI 绑定 消费变量变化事件并刷新控件；用户编辑时走独立写入链

运行时

启动流程

运行时启动流程：

StartAsync
    ↓
CommStartupService
    ↓
检查设备连接
    ↓
至少一个启用设备连接成功
    ↓
准备并启动全局脚本
    ↓
启动 PollingUpdater
    ↓
进入 Running 状态

启动时必须同时满足：

  • 至少有一个启用设备
  • 启用设备下至少有一个启用通道
  • 通道已绑定结构体并配置了地址
  • 至少一个设备连接成功

如果没有可用设备、没有可用通道，或所有设备连接失败：

启动失败
Runtime 保持 Stopped

停止流程

运行时停止流程：

StopAsync
    ↓
取消轮询任务
    ↓
清理运行时资源
    ↓
状态恢复为 Stopped

变量轮询

任务模型

轮询器按 DeviceId 分组，每台设备创建一个 Task.Run：

设备 A -> Task A -> Channel A1 -> Channel A2 -> Delay
设备 B -> Task B -> Channel B1 -> Channel B2 -> Delay

其并发策略是：

  • 不同设备之间并行
  • 同一设备的通道之间串行

这个设计避免同一协议设备被多个轮询任务同时访问，同时允许不同设备并发通信

轮询间隔：

Math.Max(device.PollingInterval, 10)

实际相邻两轮开始时间约为：

本轮所有通道执行耗时 + PollingInterval

它不是固定频率定时器。如果读取耗时增加，实际轮询周期也会变长

轮询流程

PollingUpdater
    ↓
CommService.ReadStructureAsync
    ↓
协议读取结构体
    ↓
VariableValueCommitter
    ↓
更新变量中心
    ↓
触发绑定刷新

读取属于单向同步：

设备 -> 变量中心

轮询只会读取满足以下条件的通道：

  • 设备已启用
  • 通道已启用
  • 通道绑定了结构体
  • 结构体存在并已启用
  • 结构体访问模式为 ReadOnly 或 ReadWrite

WriteOnly 结构体不会被轮询读取

轮询处理

  1. 在通信运行时启动时，根据当前工程的“设备 → 通道 → 结构体变量”配置构建轮询上下文
  2. 按设备创建独立后台任务
  3. 在每个设备任务中，依次处理该设备下所有可轮询通道
  4. 普通变量调用 DynamicViewCommService.ReadStructureAsync，从设备读取整个结构体
  5. 特殊用途变量按 DynamicViewVariableUsage 分派给任务处理器，例如配方任务和报警读取任务
  6. 根据本次处理结果 DynamicViewCommOutcome 更新设备状态：
    • Read：记录最近读取时间与在线状态
    • Written：记录最近写入时间与在线状态（配方等会回写设备的任务）
    • Skipped / NoData：不可通信而跳过，或未读到数据，不计为成功
    • 异常：记录错误状态

报警任务处理器与普通读取一样调用整体结构体读取，但只处理 Usage=Alarm 的只读 Bool 结构体。读取结果提交变量中心后，由报警运行时服务检测触发与恢复；通讯线程本身不直接更新报警列表或数据库。详细规则参见报警系统。

ReadStructureAsync 在上下文不可通信时返回 Skipped，读取到空数据时返回 NoData。跳过读取不会再被误记为读取成功

轮询的基本单位不是单个字段，而是一个 DynamicViewCommunicationChannelContext：

设备 Device + 通道 Channel + 结构体变量 Structure

普通变量最终调用 Modbus ReadStructAsync，按照通道配置的地址区和起始地址读取整个运行时结构体

异常范围

异常捕获以单个通道为单位，单通道异常不会影响同设备本轮的其它通道

某个通道抛出异常后，按设备连接状态分两种情况处理：

通道级错误（连接仍正常）：

  1. 记录该通道的错误信息
  2. 继续遍历本设备的后续通道
  3. 下一轮仍会重新尝试该通道

因此排序靠前的通道持续失败，不会导致后续通道长期得不到轮询

设备断线（连接已断开）：

  1. 中断本设备本轮剩余通道
  2. 调用 DynamicViewCommService.TryReconnectDeviceAsync 尝试重连，重连次数与等待时间取自协议配置的 MaxRetries 与 ExecutionTimeout
  3. 重连成功：恢复在线状态，下一轮继续轮询
  4. 重连失败：将该设备标记为故障并停止其轮询任务，不再持续刷错误日志

故障与异常停止

当所有设备都因断线且重连失败而进入故障状态时，轮询器触发 IDynamicViewVariablePollingUpdater.Faulted 事件。

DynamicViewCommRuntimeService 订阅该事件后：

  1. 触发 OnTaskAbnormalStop（任务名“动态视图通讯运行时异常停止”）
  2. 取消运行时令牌，经 KeepAliveAsync → CleanupAsync 正常收尾，将运行时状态切换为 Stopped

仅当所有设备均故障时才会异常停止整个运行时。部分设备故障时，故障设备会停止轮询（不再刷日志），其余设备继续正常运行；故障设备需重启运行时才会重新连接。建议在配置阶段禁用不可用的设备

多设备架构

通讯系统支持：

  • 多 PLC
  • 多 IP
  • 多端口
  • 多协议
  • 分布式设备

每个设备：

  • 独立连接
  • 独立状态
  • 独立轮询
  • 独立异常处理

设备之间互不影响

轮询按设备分组执行。同一设备内的通道按顺序读取，不同设备之间可以并行轮询。设备轮询间隔来自设备配置，实际运行时最小不会低于 10 ms

通讯状态系统

通讯状态系统仅负责：

  • 在线状态反馈
  • 最后通讯时间
  • 最后错误信息
  • UI状态显示

状态系统：

不参与变量写入权限控制
不参与工业安全逻辑
不保证实时一致性

安全控制由DynamicViewVariableWriteService负责

安全写入流程

变量写入流程：

UI操作
    ↓
VariableWriteService
    ↓
写入设备
    ↓
全部设备写入成功
    ↓
提交变量中心

如果任意设备写入失败：

变量中心不会更新

这样可保证：

界面状态 == 设备真实状态

写入只会发送到满足以下条件的通道：

  • 设备已启用
  • 通道已启用
  • 通道绑定到目标结构体
  • 通道地址不为空
  • 目标结构体已启用，且访问模式不是 ReadOnly

如果结构体没有关联任何可写通道，写入会被阻止。写入成功后，变量中心才会提交本地值并刷新界面

Modbus TCP 通道规则

当前支持的 Modbus 地址区：

地址区 读写能力 推荐字段类型 典型用途
Coils 读写 Bool 启停命令、复位信号、布尔输出
DiscreteInputs 只读 Bool 到位信号、报警输入、传感器状态
HoldingRegisters 读写 Int16、UInt16 参数设定、配方数据、工艺数值
InputRegisters 只读 Int16、UInt16 采集值、测量值、设备状态字

通道地址表示结构体读写的起始地址，必须是有效的无符号短整型。结构体字段顺序会影响实际读写顺序，实施时应与设备寄存器布局保持一致

访问模式与地址区需要兼容：

  • ReadOnly 和 ReadWrite 结构体可绑定到只读或可写地址区
  • WriteOnly 结构体只能绑定到 Coils 或 HoldingRegisters
  • 配方结构体当前只支持 HoldingRegisters
  • 报警结构体字段必须为 Bool，因此应使用 Coils 或 DiscreteInputs

结构体通讯

通讯系统采用结构体变量作为核心通讯模型

例如：

Temperature
├── CurrentValue
├── Alarm
├── IsEnabled
└── ErrorCode

读取时：

PLC结构体 → Runtime Struct → 字段字典 → 变量中心

写入时：

变量中心 → Runtime Struct → PLC结构体

结构体通讯的核心原则是“按结构体整体读写”。因此，字段新增、删除或调整顺序后，需要同步确认 PLC 地址布局，避免字段错位

线程模型

  • 通讯系统支持多设备并行轮询
  • 不同设备以独立任务运行
  • 变量中心负责运行时值线程安全
  • 通讯层不负责变量锁控制

常见问题

Note

为什么启动后没有进入 Running 状态

请检查是否存在启用设备、启用通道、通道地址、结构体绑定，以及至少一个设备能连接成功

Note

为什么界面操作后变量没有变化

写入失败时变量中心不会提交新值。请检查结构体是否为 ReadOnly、是否有关联可写通道、设备是否在线、地址区是否支持写入

Note

为什么修改设备中心后运行时没有变化

设备中心使用编辑副本。修改后需要执行检查并同步工程数据到内存
