MAS.DynamicViews.ProjectModel

工程模型是 MAS.DynamicViews 的数据根节点

所有页面、窗口、背景、控件、结构体变量、通讯设备、通讯通道、参数和全局脚本，最终都应归属于工程模型

工程模型的核心目标是：

  • 描述完整动态视图工程
  • 支持工程持久化
  • 支持设计态编辑
  • 支持运行态加载
  • 支持页面、变量、设备之间的关联
  • 支持后续扩展和版本迁移

工程模型

DynamicViewProjectModel用于描述一个完整动态视图工程

  • 保存工程基础信息
  • 保存页面集合
  • 保存窗口集合
  • 保存背景集合
  • 保存结构体变量集合
  • 保存通讯设备集合
  • 保存内容分组
  • 保存项目参数
  • 保存全局脚本代码
  • 保存启动页面
  • 保存统一页面尺寸
  • 作为设计态和运行态的共同数据源

常见属性包括但不限于：

ProjectId
Name
StartupPageId
PageWidth
PageHeight
Pages
Windows
Backgrounds
Structures
Parameters
CommunicationDevices
ContentGroups
GlobalScriptCode

页面模型

DynamicViewPageModel用于描述一个动态页面

  • 页面名称
  • 页面标题
  • 页面尺寸
  • 页面外观
  • 页面控件集合

常见属性包括但不限于：

DynamicViewPageModel
├── PageId
├── Name
├── Title
├── Width
├── Height
├── Appearance
└── Controls

页面只是描述当前界面应该包含哪些控件，以及这些控件如何布局

窗口模型

窗口用于在运行时弹出独立界面。适合参数确认、设备详情、报警处理、配方操作等不适合放在主画面的内容

窗口常见配置包括：

  • 窗口名称和标题
  • 宽度和高度
  • 是否允许调整大小
  • 是否模态显示
  • 是否允许重复打开
  • 是否显示最大化、最小化按钮
  • 窗口内控件集合

运行时打开窗口时，如果窗口不允许重复打开，再次触发打开动作会激活已有窗口，而不是创建新窗口

背景模型

背景用于复用公共画面内容，例如：

  • 顶部标题栏
  • 底部状态栏
  • 左侧导航栏
  • 设备运行摘要区
  • 公共报警提示区

页面可以选择一个背景。运行时显示页面时，会先加载页面关联的背景，再显示页面自身控件。适合多个页面共享统一框架，减少重复维护

页面尺寸

页面尺寸应优先由工程统一管理

Project.PageWidth  → 默认页面宽度
Project.PageHeight → 默认页面高度
  • 新增页面时，应使用工程级页面尺寸作为默认值
  • 当工程页面尺寸变化时，可以按比例同步缩放已有页面中的控件布局
  • 这样可以避免每个页面尺寸不一致导致运行时显示混乱

控件模型

DynamicViewControlModel用于描述页面中的单个控件

常见属性包括但不限于：

DynamicViewControlModel
├── ControlId
├── Name
├── ControlType
├── Properties
├── Bindings
├── Events
└── Security

控件模型主要包含三类配置：

配置 说明
Properties 控件布局、文本、外观等静态属性
Bindings 控件属性与变量字段的绑定关系
Events 控件运行时事件配置
Security 控件运行时操作保护配置

控件属性通常通过DynamicViewControlPropertiesModel统一管理

常见属性包括但不限于：

Properties

└── Common

布局属性用于描述控件在页面中的位置和尺寸

常见字段：

外观属性用于描述控件视觉样式

常见字段：

Background

控件绑定模型

控件绑定用于描述控件属性与变量字段之间的关系

典型结构：

Bindings
├── Text
├── Visibility
├── IsEnabled
├── Color
├── Checked
├── Status
└── Value

绑定模型不应直接保存变量对象引用，而应保存稳定标识：

VariableId
FieldId
DisplayName

其中：

字段 说明
VariableId 结构体变量标识
FieldId 字段标识
DisplayName UI 展示名称，例如 Motor.Speed

DisplayName仅用于界面展示，不应作为业务关联依据

控件事件模型

控件事件用于描述运行时交互行为

典型结构：

Events
├── Click
├── MouseDown
└── MouseUp

事件模型只描述事件配置，不直接执行逻辑。真正的执行由运行时处理。每个事件可配置切换页面、打开窗口和变量操作

适用建议：

  • Click：适合启动、停止、复位、打开窗口、切换页面等一次性动作
  • MouseDown：适合按下立即动作、点动、长按保护等场景
  • MouseUp：适合释放保持信号、结束点动、撤销临时控制等场景

变量模型

DynamicViewVariableModel用于描述动态视图中的结构体变量

当前推荐采用结构体变量 + 字段集合模型

典型结构：

DynamicViewVariableModel
├── VariableId
├── Name
├── GroupName
├── AccessMode
├── IsEnabled
├── Usage
├── Fields
├── Description
└── Properties

字段模型

字段模型用于描述结构体变量中的单个字段

典型结构：

DynamicViewVariableFieldModel
├── FieldId
├── Name
├── VariableType
├── DefaultValue
├── FixedString
└── Description

字段是控件绑定的最小单位

例如：

Motor
├── Running
├── Speed
├── Alarm
└── Temperature

控件绑定时，应绑定到Motor.Speed而不是只绑定到Motor

设备模型

设备模型用于描述外部通讯设备

典型结构：

DynamicViewCommunicationDeviceModel
├── DeviceId
├── Name
├── IsEnabled
├── ProtocolType
├── ModbusTcpConfig
├── PollingInterval
└── Channels

当前设备中心支持 Modbus TCP。设备模型保存连接参数和通道集合，不直接处理通讯逻辑。真实通讯由运行时通讯服务负责

通道模型

通道模型用于描述设备数据区与变量字段之间的映射关系

典型结构：

DynamicViewCommunicationChannelModel
├── ChannelId
├── Name
├── IsEnabled
├── AddressArea
├── Address
├── StructureId
├── PollingIntervalOverride
├── DisplayName
└── Description

PollingIntervalOverride 为可空毫秒值：null 表示继承设备的 PollingInterval，有值时仅用于延长该通道的轮询周期。设备周期是最快周期，配置值短于设备周期时仍按设备周期执行

通道绑定结构体变量时保存StructureId，而不是直接保存变量对象，这样可以：

  • 降低序列化复杂度
  • 避免对象引用失效
  • 支持引用追踪
  • 支持删除检查
  • 支持后续迁移

通道关联的是整个结构体，而不是单个字段。Modbus 运行时会从起始地址开始按结构体字段顺序读写

参数模型

参数模型用于保存工程级配置，可用于：

  • 默认页面尺寸
  • 运行时配置
  • 通讯配置
  • UI 配置
  • 设计器配置
  • 未来扩展参数

参数模型应尽量保持稳定，避免频繁破坏旧工程兼容性

模型引用原则

工程模型内部推荐使用标识引用，而不是对象引用

推荐：

VariableId
FieldId
PageId
ControlId
DeviceId
ChannelId

而不是直接保存对象引用，因为：

  • 方便序列化
  • 方便持久化
  • 方便跨模块引用
  • 方便引用追踪
  • 方便删除校验
  • 方便版本迁移

显示名称原则

显示名称只用于 UI 展示

例如：

Motor.Speed
Device1.Channel1
Page1.Button1

显示名称不应作为持久化关联依据，原因：

  • 名称可能被用户修改
  • 名称可能重复
  • 名称可能本地化
  • 名称不适合作为稳定标识

稳定关联必须依赖唯一 Id

内容同步原则

数据中心和设备中心采用“编辑副本 → 校验 → 同步工程数据到内存”的方式工作

这意味着：

  • 在窗口中修改结构体、字段、设备或通道后，必须同步到工程数据，运行时才会使用新配置
  • 关闭窗口时如果存在未同步改动，会提示是否保存
  • 同步前会进行校验；校验失败时不会写入工程
  • 删除结构体或字段前会检查引用，仍被控件绑定、控件事件或通讯通道引用时会阻止删除

设计态使用方式

设计态主要通过工程模型完成：

  • 页面展示
  • 控件创建
  • 控件选择
  • 属性编辑
  • 绑定配置
  • 事件配置
  • 页面尺寸调整
  • 历史记录保存
  • 脏状态标记

设计态修改模型后，应通知持久化系统标记工程已修改

运行态使用方式

运行态主要通过工程模型完成：

  • 加载启动页
  • 渲染页面控件
  • 建立变量绑定
  • 建立事件处理
  • 构建通讯映射
  • 启动轮询服务
  • 处理变量写回

运行态应优先使用启动时缓存的数据结构，避免频繁访问持久化服务

持久化原则

工程模型需要支持完整持久化

持久化时应保存：

  • 工程基础信息
  • 页面集合
  • 控件集合
  • 控件属性
  • 控件绑定
  • 控件事件
  • 变量集合
  • 字段集合
  • 设备集合
  • 通道集合
  • 项目参数

不应保存：

  • WPF 控件实例
  • 运行时服务实例
  • 临时缓存
  • 非必要 UI 状态
  • 不可序列化对象

版本兼容

工程模型后续可能持续演进，因此需要注意版本兼容

建议原则：

  • 新增字段优先提供默认值
  • 避免删除已有字段
  • 避免直接改变字段语义
  • 关键模型建议预留版本号
  • 复杂迁移应通过迁移服务完成

常见问题

Note

为什么控件模型不直接保存 WPF 控件

因为控件模型需要持久化，而 WPF 控件实例不可稳定序列化

正确方式是：

  • ControlModel → Renderer → WPF Control
Note

为什么绑定保存 VariableId 和 FieldId

因为变量名称和字段名称可能被修改

使用 Id 可以保证绑定关系稳定
