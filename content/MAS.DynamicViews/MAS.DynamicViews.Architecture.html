MAS.DynamicViews.Architecture

MAS.DynamicViews 采用设计态与运行态分离的动态组态架构

整个系统围绕：

  • 工程模型
  • 页面模型
  • 窗口和背景模型
  • 控件模型
  • 变量系统
  • 通讯系统
  • 渲染系统
  • 运行时系统
  • 控件安全
  • 全局脚本

进行协同构建

系统目标是：

  • 支持工业动态组态
  • 支持运行时数据驱动
  • 支持 PLC 通讯联动
  • 支持可扩展控件体系
  • 支持 MVVM 架构
  • 支持后续插件化扩展

设计理念

模型驱动

MAS.DynamicViews采用模型驱动架构

页面、控件、变量、设备、绑定、事件都优先以模型描述

渲染器只负责：模型 → WPF 界面元素，而不是直接操作业务逻辑

这样可以实现：

  • 设计态与运行态统一
  • 工程可持久化
  • 动态生成界面
  • 后续跨平台渲染
  • 运行时热加载

设计态与运行态

系统核心原则之一是：

  • 设计态负责编辑
  • 运行态负责执行

设计态

设计态主要用于：

  • 控件编辑
  • 页面布局
  • 拖拽创建
  • 属性修改
  • 事件配置
  • 对齐辅助
  • 缩放操作
  • 历史记录

设计态强调：

  • 编辑体验
  • 可视化操作
  • 布局精确性
  • 非侵入式选择框

运行态

运行态主要用于：

  • 实际界面显示
  • 变量绑定
  • 数据刷新
  • PLC 联动
  • 用户交互
  • 运行时事件执行

运行态强调：

  • 数据驱动
  • 稳定性
  • 性能
  • 实时性

工程模型

整个系统的数据入口是：DynamicViewProjectModel

工程模型通常包含：

Project
├── Pages
├── Windows
├── Backgrounds
├── Structures
├── CommunicationDevices
├── Parameters
└── GlobalScriptCode

页面模型

页面用于描述动态界面

每个页面通常包含：

Page
├── Appearance
├── Controls
├── Width
├── Height
└── Background

页面本身不直接依赖运行时逻辑

控件模型

控件模型用于描述动态控件

每个控件通常包含：

Control
├── Layout
├── Appearance
├── Text
├── Bindings
└── Events

控件模型只描述状态，不直接持有真实 WPF 控件实例

渲染系统

渲染系统负责：模型 → WPF 控件

系统当前包含：

DynamicViewRendererBase
├── DynamicViewDesignerRenderer
└── DynamicViewRuntimeRenderer

设计态渲染器

设计态渲染器主要负责：

  • 创建设计态控件
  • 添加选择框
  • 添加缩放控制点
  • 提供设计辅助能力

设计态通过DesignerControlChrome包装真实控件。这样可以避免：

  • 修改真实控件尺寸
  • 影响布局测量
  • 影响文本排列
  • 影响运行态表现

运行态渲染器

运行态渲染器主要负责：

  • 创建真实运行控件
  • 建立运行时绑定
  • 绑定运行事件
  • 连接运行时上下文

运行态渲染器会与以下协同工作：

  • RuntimeBindingManager
  • RuntimeEventExecutor
  • VariableService

变量系统

变量系统是动态视图的数据核心，当前系统采用结构体变量 + 字段绑定模式

结构体变量

结构体变量用于描述：

  • PLC 数据块
  • 设备数据结构
  • 运行时状态集合

例如：

Motor
├── Running
├── Speed
├── Alarm
└── Temperature

字段级绑定

大部分控件绑定是结构体字段，而不是单独的全局变量

例如：

  • Motor.Speed
  • Motor.Alarm

这样可以：

  • 更符合 PLC 数据组织
  • 更方便通讯映射
  • 更方便批量同步
  • 更方便引用追踪

绑定系统

绑定系统负责：变量 → 控件属性

常见绑定包括：

  • 文本绑定
  • 状态绑定
  • 数值绑定
  • 可见性绑定
  • 启用状态绑定
  • 颜色绑定
  • 选中状态绑定

绑定系统统一通过Binding Model描述，而不是每个控件单独实现绑定逻辑

通讯系统

通讯系统负责PLC ↔ DynamicViews之间的数据同步

核心职责

  • 设备管理
  • 通道管理
  • 数据轮询
  • 数据写回
  • 协议适配
  • 变量同步

数据轮询

轮询系统负责PLC → VariableService运行时更新

轮询系统会缓存：

  • 设备
  • 通道
  • 变量映射

避免频繁读取工程数据

数据回写

写回系统负责VariableService → PLC同步

主要用于：

  • 按钮操作
  • 开关控制
  • 参数写入
  • 用户交互

运行时系统

运行时系统负责：

  • 页面运行
  • 数据刷新
  • 用户交互
  • 事件执行

核心组件包括：

Runtime
├── Runtime Renderer
├── Runtime Binding Manager
├── Runtime Event Executor
├── Runtime Context
└── Runtime Host

事件系统

事件系统用于描述用户操作 → 动作执行

例如：

  • 点击按钮
  • 按下按钮
  • 松开按钮
  • 页面跳转
  • 打开窗口
  • 写入变量

事件系统与Runtime Context协同工作，避免设计态与运行态逻辑混合

当前控件事件支持点击、按下和松开。每个事件可配置切换页面、打开窗口和变量操作。

控件安全

控件安全用于在运行态执行事件前进行保护，主要包括：

  • 最低角色校验
  • 按下事件长按限制
  • 按下事件防抖
  • 二次确认
  • 操作日志

权限不足时事件不会执行。启用操作日志后，系统会记录操作者、事件类型、资源和耗时。

全局脚本

全局脚本是工程级运行时扩展点。当前只支持全局脚本，不支持页面脚本或控件事件脚本。

脚本可以通过公开服务访问变量、变量写入、导航和动态视图管理能力。脚本编译、初始化或启动失败会导致动态视图运行时启动失败。

持久化系统

持久化系统负责：

  • 工程保存
  • 工程恢复
  • 脏状态跟踪
  • 页面变更监听
  • 变量变更监听

系统目标是完整恢复工程运行状态

历史系统

历史系统用于支持：

  • Undo
  • Redo

当前支持：

  • 控件创建
  • 控件删除
  • 控件移动
  • 属性修改
  • 页面操作

历史系统主要服务于设计器

引用追踪

引用追踪系统用于变量 ↔ 控件 ↔ 通讯 ↔ 事件之间的关系分析

主要用于：

  • 删除检查
  • 重命名同步
  • 调试排查
  • 引用查看
  • 工程维护

这是后期大型工程维护的重要基础能力

扩展架构

系统保留扩展能力，包括：

  • 自定义控件
  • 自定义预设
  • 自定义绑定
  • 自定义事件
  • 自定义渲染器
  • 插件扩展

生命周期

系统典型运行流程：

加载工程
    ↓
加载页面模型
    ↓
创建渲染器
    ↓
生成运行控件
    ↓
建立变量绑定
    ↓
绑定运行事件
    ↓
启动全局脚本
    ↓
启动通讯轮询
    ↓
运行时交互
