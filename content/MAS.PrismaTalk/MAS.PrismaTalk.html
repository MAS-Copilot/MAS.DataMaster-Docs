MAS.PrismaTalk

MAS.PrismaTalk 是 MAS DataMaster（DM）专用的单 Agent 对话模块。它不是独立的通用 Agent 平台，也不负责多 Agent 编排。它的目标是让一个模型在统一、安全的运行时中理解用户请求，调用 DM 与扩展工具，并把执行过程和最终结果呈现给用户。

产品定位

PrismaTalk 负责：

  • 维护一次会话的模型上下文和跨轮状态；
  • 调用模型并处理流式响应；
  • 汇聚、发现和执行多来源工具；
  • 对高风险操作发起用户审批；
  • 展示自然语言回复、工具时间线和错误状态；
  • 在长对话中评估上下文预算并压缩较早历史。

数据库、资产、设备和其他依赖 DM 运行环境的能力由宿主提供。PrismaTalk 通过 NamedPipe IPC 使用这些能力；MCP 用于第三方或客户自定义扩展，不是 DM 内部能力的主干。

架构概览

系统按职责分为四个边界：

边界 职责
会话 保存模型推理历史、已激活工具、调用幂等状态、token 用量和压缩状态
Agent Runtime 执行“模型 → 工具 → 结果 → 模型”的循环，决定何时完成一次对话
适配与执行 把领域请求映射为具体模型 API；聚合工具、执行审批和调用具体 provider
展示 WebView 展示用户消息、流式回复、工具时间线、审批和重连状态

窗口层只管理输入、发送、取消、新会话和窗口显示状态。它不保存模型上下文，也不复制维护一份消息历史。一次调用仍返回结构化结果，供测试、自动化或未来非 WebView 调用方使用。

一次请求的生命周期

  1. 用户输入被追加到当前会话。
  2. Runtime 按当前配置重建系统提示、Skill 目录和工具发现提示。
  3. 工具注册表汇聚内置工具、宿主 IPC 工具和已配置的 MCP 工具。
  4. 当前领域上下文和活跃工具被组装为统一模型请求。
  5. API 适配器将请求转换为 Chat Completions 或 Responses 请求体并发送。
  6. 自然语言增量立即进入 WebView；结构化工具调用不会作为正文展示。
  7. 工具执行管线解析真实工具名、检查风险、必要时等待审批，然后调用对应 provider。
  8. 结构化工具结果写回会话并再次发送给模型。
  9. 当模型不再调用工具时，最终回答写入会话并结束本轮。

循环最多允许 99 个工具编排步骤。达到上限后，Runtime 会移除工具并要求模型直接总结当前结果。

与 DM 宿主的关系

DM 宿主是业务能力执行环境，PrismaTalk 是对话和 Agent 编排器。二者通过固定 NamedPipe IPC 通道通信：

  • 启动时先调用 ipc.ping 检查宿主通信是否可用；
  • IPC 不可用并不等价于宿主进程一定未启动，也可能是 IPC 服务尚未就绪；
  • 用户可以选择继续使用不依赖宿主的能力，或者关闭 PrismaTalk；
  • 宿主工具通过 tool.list 进入统一目录，通过 tool.invoke 执行；
  • 宿主工具和本地/MCP 工具使用相同的风险、审批与结果协议。

模型通道

Runtime 使用统一的领域请求和响应，不直接依赖某一种模型 API。当前支持：

通道 请求形态 工具形态
Chat Completions messages assistant tool_calls + role=tool 结果
Responses input items function_call + function_call_output

Responses 默认采用无状态方式：每轮发送完整上下文并设置 store: false。会话状态始终由 PrismaTalk 自己维护。

工具来源

工具统一表示为名称、说明、JSON Schema、风险等级和 provider 标识。当前来源包括：

  • DM 宿主 IPC typed tools；
  • Runtime 和工作区信息工具；
  • Skill 资源读取；
  • 草稿管理；
  • Agent 计划交互；
  • C#、Python 脚本；
  • CLI 与文件操作；
  • stdio MCP server。

内部 DM 能力优先设计为 typed tools。CLI 和脚本属于兼容或兜底能力，不应替代稳定的业务接口。

设计原则

  • 使用原生 function calling，不从助手正文解析动作标签或 JSON。
  • 领域会话与具体 wire 协议分离，同一套 Runtime 可切换模型通道。
  • 模型上下文、WebView 展示和窗口状态各自只有一个责任边界。
  • 工具执行统一经过发现、风险判断、审批、调用和结果标准化管线。
  • 内部能力优先使用 IPC typed tools；MCP 只作为外部扩展入口。
  • 普通问答保持轻量，只有真实工具行为才创建可审计时间线。
  • 长对话在预算阈值处压缩较早历史，当前任务和最近历史保持原样。
