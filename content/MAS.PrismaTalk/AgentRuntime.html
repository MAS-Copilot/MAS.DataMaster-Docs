Agent Runtime

Agent Runtime 是 MAS.PrismaTalk 的核心执行层。它只编排一个 DM 专用 Agent，通过结构化 function calling 完成“模型判断 → 调用工具 → 观察结果 → 继续推理 → 最终回答”的循环。

分层与依赖方向

Runtime 内部使用领域对象表达对话和工具行为，模型 API、工具来源和 WebView 都位于边界之外：

用户输入
  → 会话驱动
  → Agent 编排循环
      → 模型 API 适配器 → HTTP/SSE
      → 工具执行管线 → 工具注册表 → Tool Provider
      → 事件与流式视图 → WebView
  → 结构化调用结果

核心规则是：编排循环不构造 HTTP 请求体、不调用 JavaScript、不直接执行某类业务工具。它只处理领域请求、领域响应和领域会话项。

领域会话项

会话历史由以下 item 组成：

Item 含义
SystemMessageItem 系统指令；通常作为每轮前置提示动态构建
UserMessageItem 用户输入或系统生成的会话摘要
AssistantMessageItem 助手说明文字或最终回答
FunctionCallItem 模型发起的工具调用，包含 CallId、工具名和 JSON 参数
FunctionCallOutputItem 与 CallId 配对的结构化工具结果

编排层不会直接操作 messages、tool_calls、function_call_output 等 wire 对象。协议映射只发生在模型适配边界。

统一模型请求与响应

一次模型调用使用统一领域请求：

ModelRequest
├─ Items                 当前完整领域上下文
├─ Tools                 本次允许模型调用的工具
├─ Model?                可按请求覆盖模型
├─ PreviousResponseId?   可选的 Responses 关联标识
├─ MaxOutputTokens?      最大输出 token
└─ Temperature?          采样温度

一次模型响应使用统一领域结果：

ModelTurn
├─ Items          助手文本和零到多个工具调用
├─ ResponseId?    Responses 通道可返回
├─ FinishReason   结束原因
└─ Usage?         模型报告的 token 用量

一个 turn 可以同时包含助手说明文字和多个工具调用。说明文字流向用户，工具调用进入执行管线，二者不会混在同一个文本解析通道中。

编排循环

每次用户请求按以下顺序运行：

  1. 获取当前会话，并修复上一轮取消或异常留下的悬空工具调用。
  2. 汇聚工具目录，恢复本会话已经激活的隐藏工具。
  3. 动态构建本轮前置上下文：系统提示、工具发现规则和 Skill 目录。
  4. 评估下一次模型请求的上下文预算，必要时先压缩较早历史。
  5. 将前置上下文、会话历史和活跃工具组装为 ModelRequest。
  6. 模型输出自然语言时实时追加到当前流式消息。
  7. 模型输出工具调用时，把助手说明和调用 item 写入会话。
  8. 逐个执行工具调用，把每个结果 item 写入会话。
  9. 带着工具结果再次调用模型。
  10. 模型不再调用工具时，把最终回答写入会话并完成本轮。

同一个 CallId 在会话中只允许执行一次。模型重复使用已经执行的调用标识时，Runtime 不会再次触发副作用，而是回填重复调用错误。

工具发现与激活

工具采用两层目录：少量基础工具直接处于活跃集，大量业务工具通过发现元工具按需检索和激活。这样可以避免把全部 JSON Schema 放进每次请求，降低 token 消耗，并减少模型在相似工具之间误选。

工具激活状态属于会话状态。某个隐藏工具在本轮激活后，后续轮次会自动恢复；新会话不会继承旧会话的激活状态。

真实工具名可能包含点号或超过模型 API 的函数名限制。发送前会映射为 wire-safe 名称，执行时再映射回真实名称。工具注册表和业务 provider 始终使用真实名称。

工具执行管线

每个工具调用统一经过以下步骤：

  1. 将 wire-safe 名称解析为真实工具名；
  2. 必要时自动激活目录中的目标工具；
  3. 解析工具定义、provider 和风险等级；
  4. 找不到工具时返回可供模型修正的结构化错误；
  5. 高风险工具执行前请求用户审批；
  6. 路由到具体 provider；
  7. 将成功或失败结果标准化为 ToolResult；
  8. 发送开始、等待审批、完成等时间线事件；
  9. 生成与原 CallId 配对的输出 item。

工具结果的领域形态为：

{
  "isSuccess": true,
  "summary": "操作完成",
  "data": "可选的结构化或文本结果",
  "error": null
}

映射到模型协议时，结果会被序列化为稳定 payload。模型收到的是执行事实，不是 WebView 展示事件。

风险与审批

风险 当前行为
Low 直接执行
Medium 直接进入 provider，由工具继续进行参数和业务校验
High 执行前等待用户审批

高风险审批支持“本次允许”和“本轮始终允许”。轮级允许状态只在当前用户请求内有效，下一次用户请求开始时重置。用户拒绝不会抛弃调用历史，而是生成失败工具结果，让模型能够解释取消原因或选择其他方案。

Chat Completions 映射

领域请求映射为 OpenAI 兼容 Chat Completions 请求：

{
  "model": "model-name",
  "messages": [
    { "role": "system", "content": "..." },
    { "role": "user", "content": "..." },
    {
      "role": "assistant",
      "content": "我先检查当前状态。",
      "tool_calls": [
        {
          "id": "call_1",
          "type": "function",
          "function": {
            "name": "runtime_workspace",
            "arguments": "{}"
          }
        }
      ]
    },
    {
      "role": "tool",
      "tool_call_id": "call_1",
      "content": "{\"isSuccess\":true,\"summary\":\"已获取工作区\",\"data\":\"...\"}"
    }
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "runtime_workspace",
        "description": "Get current workspace information",
        "parameters": { "type": "object", "properties": {} }
      }
    }
  ],
  "stream": true
}

SSE 中的 choices[].delta.content 作为自然语言增量；choices[].delta.tool_calls 按索引累计名称和参数，完成后转换为领域工具调用。

Responses 映射

Responses 通道把同一批领域 item 映射为 input：

{
  "model": "model-name",
  "input": [
    {
      "role": "user",
      "content": "检查当前工作区"
    },
    {
      "type": "function_call",
      "call_id": "call_1",
      "name": "runtime_workspace",
      "arguments": "{}"
    },
    {
      "type": "function_call_output",
      "call_id": "call_1",
      "output": "{\"isSuccess\":true,\"summary\":\"已获取工作区\",\"data\":\"...\"}"
    }
  ],
  "tools": [
    {
      "type": "function",
      "name": "runtime_workspace",
      "description": "Get current workspace information",
      "parameters": { "type": "object", "properties": {} }
    }
  ],
  "store": false,
  "stream": true
}

Responses 默认不依赖服务端会话存储：每轮发送完整 input，并设置 store: false。previous_response_id 只有在领域请求显式提供时才透传。

SSE 按事件 type 分发，文本增量和函数调用分别累计；completed、incomplete 和 [DONE] 归一化为终态，failed 或 error 转为连接异常交由 Runtime 判断是否重连。

会话状态

每个会话维护：

状态 用途
Items 完整领域历史，包括用户、助手、工具调用和工具结果
ActivatedToolNames 已按需激活的隐藏工具
已执行 CallId 防止重复执行同一个工具调用
最近 Usage 优先使用模型报告的真实 token 用量
ContextWindow 当前配置的上下文窗口
CompactionPending 是否应在下一轮开始前压缩

当前会话存储位于进程内。创建新会话会丢弃旧会话状态并生成新的会话标识。

上一轮若在工具调用写入后、结果写入前被取消，下一轮开始时会在原调用块后补入“上一轮已中断”的失败结果，保证每个调用都有配对输出，满足模型协议约束。

上下文预算与压缩

可用输入预算按以下思路计算：

可用输入预算 = 上下文窗口 - 最大输出 token - 安全余量
  • 达到约 70% 的软阈值时，标记下一轮开始前压缩；
  • 达到约 90% 的硬阈值时，在发送请求前立即尝试压缩；
  • 有真实 usage 时，以最近 usage 为基准，并估算之后新增的 item；
  • 没有真实 usage 时，对前置提示和完整会话进行估算。

压缩会保留当前用户任务，把更早的完整前缀替换为一条 CONVERSATION_SUMMARY。摘要最多使用上下文窗口的 10%，再次压缩时旧摘要会与后续历史合并重写，不产生嵌套摘要。

当前摘要输入只包含系统、用户和助手文本，不包含工具调用参数及原始工具结果。这能控制敏感数据和 token 体积，但意味着只存在于工具结果、未被助手复述的事实可能无法进入摘要；开发业务工具和长任务流程时应注意这一边界。

重连与取消

连接中断时，Runtime 最多重试 5 次，每次等待 10 秒。重试前会回滚当前未完成的助手流式输出并显示重连状态，成功或最终失败后隐藏状态。

用户取消会停止模型请求、工具调用等待或重连延迟。已经写入但尚未配对结果的工具调用由下一轮的历史修复机制处理。

展示边界

WebView 接收两类输出：

  • 流式视图：用户可见的助手文本、重连回滚和重连提示；
  • Agent 事件：思考、工具开始、等待审批、工具完成、上下文用量和本轮完成。

WebView 不决定工具是否允许执行，也不保存模型推理历史。审批结果只是用户输入，最终风险判断和工具调用始终在后端执行
