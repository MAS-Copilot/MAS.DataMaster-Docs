Agent Runtime

Agent Runtime 是 MAS.PrismaTalk 当前的核心执行层。它把“模型回答”扩展为“模型可以结构化调用工具、观察结果、继续推理并最终回答”的循环

运行流程

一次对话由 ConversationDriver 进入 AgentLoop：

  1. 读取系统提示词和当前对话历史
  2. 通过 ToolRegistry.ListAllAsync 汇聚所有工具定义
  3. 根据当前 IAgentProtocol 生成模型可理解的工具上下文或 tools schema
  4. 调用 PrismaTalkClient 发送 OpenAI 兼容流式请求
  5. 如果模型返回工具调用，则解析参数、检查风险、必要时请求用户审批
  6. 通过 ToolRegistry.InvokeAsync 路由到具体 IToolProvider
  7. 把 ToolResult 回填给模型，继续下一轮推理
  8. 没有工具调用时，将自然语言内容作为最终回答写入 WebView

当前 MAX_STEPS 为 8，超过后会要求模型停止调用工具并总结已完成工作

协议适配

代码保留两套协议适配，但上层 loop 面向统一动作模型

协议 文件 用途
Native tool calls Agent/Protocols/NativeToolCallProtocol.cs 使用 OpenAI 兼容 tools / tool_calls，动作和文本分别走结构化通道与 content 通道
Text envelope Agent/Protocols/TextEnvelopeProtocol.cs 端点不支持 tool_calls 时的 fallback，模型输出 <agent_action>{...}</agent_action>

当前依赖注册中默认使用 NativeToolCallProtocol。该协议会把真实工具名映射为 wire-safe 名称，例如 entity.listInstances 或 mcp.example.search 会在发送给模型前转换为仅含字母、数字、下划线或短横线的函数名，并在执行时映射回真实工具名

结构化消息

Native 协议下，工具调用不再混入正文：

  • 助手自然语言输出走 delta.content，会实时进入 .stream-output
  • 工具调用走 delta.tool_calls，由 AssistantStreamResult.ToolCalls 累积
  • 工具结果写回 role=tool 消息，并通过 tool_call_id 与原调用关联
  • 对话历史中的 assistant 工具调用保存为 ChatMessage.ToolCalls

这种结构避免旧文本信封模式下把 JSON、标签或半截工具参数泄漏给用户

工具风险与审批

每个 ToolDefinition 都带有 ToolRisk：

风险 当前行为
Low 直接执行
Medium 当前代码直接执行，工具自身仍应做参数校验
High 执行前调用 IToolApprovalService.RequestApprovalAsync

默认审批实现是 WebViewToolApprovalService。它调用前端 renderAgentApprovalRequest 渲染审批卡片，并等待前端回传 toolApprovalResponse。用户拒绝时，Agent 会收到一个失败的 ToolResult，模型可据此解释已取消或选择其他方案

Skill 资源读取

早期设计中的 resource_request 在 native tool_calls 模式下被收敛为标准工具 skill.read。SkillResourceToolProvider 读取 FileSystemSkillRegistry 中发现的 SKILL.md 或资源文件，并以低风险工具形式进入统一目录

FileSystemSkillRegistry 从 ApplicationInformation.PRISMATALK_PATH 读取 Skill 包，解析 front matter 中的 name、description、risk，并索引 references/、examples/、assets/、templates/ 等资源
