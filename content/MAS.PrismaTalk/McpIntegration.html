MCP 集成

MAS.PrismaTalk 已经支持 stdio MCP server

当前支持范围

当前 MCP provider 支持：

  • 按 PrismaTalkConfig.McpServers 读取 server 列表
  • 启动本地 stdio 进程
  • 发送 initialize
  • 发送 notifications/initialized
  • 调用 tools/list 发现工具
  • 调用 tools/call 执行工具
  • 将 MCP 工具暴露为 PrismaTalk 统一工具目录
  • 使用统一 ToolRisk 和 WebView 审批门

因此 MCP 目前适合作为高级配置能力，而不是普通用户默认功能

配置模型

McpServerConfig 字段如下：

字段 说明
Enabled 是否启用该 server
Name 本地唯一名称，也会参与工具名前缀
Command 可执行命令，例如 node、npx、uvx 或完整路径
Arguments 命令行参数
WorkingDirectory 可选工作目录
Environment 传给 MCP server 进程的环境变量
Risk 该 server 工具默认风险，默认为 high
ToolRiskOverrides 按原始工具名或暴露工具名覆盖风险
ProtocolVersion 初始化时发送的 MCP 协议版本
TimeoutSeconds 单次请求超时，最大 300 秒

示例：

{
  "McpServers": [
    {
      "Enabled": true,
      "Name": "example",
      "Command": "node",
      "Arguments": ["path/to/server.js"],
      "Risk": "high",
      "ToolRiskOverrides": {
        "safeReadOnlyTool": "low"
      },
      "TimeoutSeconds": 30
    }
  ]
}

未配置 McpServers 或列表为空时，MCP provider 返回空目录，不影响普通对话、Skill、CLI 或宿主工具

工具命名

MCP 原始工具会暴露为：

mcp.<server>.<tool>

例如 server 名为 github，原始工具名为 search_repositories，则 PrismaTalk 内部工具名为：

mcp.github.search_repositories

Native function-calling 发送给模型时会再做 wire name 映射，避免 OpenAI 兼容函数名不允许点号或超长名称的问题。执行时再映射回真实工具名

风险控制

MCP 工具默认风险是 high，这意味着每次执行前都会触发 WebView 审批卡片。只有确认某个工具是只读且安全时，才建议通过 ToolRiskOverrides 下调为 low

推荐策略：

  • 第三方 MCP server 默认 Risk = "high"
  • 只读查询类工具可以单独降为 low
  • 写文件、执行命令、访问外部系统、修改数据的工具保持 high
  • 现场自定义 server 应先在测试环境验证工具列表、参数 schema 和错误输出

与内部工具的边界

内部 MAS 软件 Agent 不需要为了“统一协议”而把所有能力做成 MCP。对于 MAS.DataMaster 自身能力，NamedPipe IPC 更合适：

  • 可以直接复用宿主 DI 和业务服务
  • 工具定义、风险、日志和权限都在同一代码体系内
  • 不需要额外进程管理和协议版本兼容
  • 异常诊断更直接

MCP 更适合作为外部扩展接口：

  • 接第三方系统
  • 接开源 MCP server
  • 接非 .NET 工具
  • 让同一工具被其他 Agent 客户端复用
  • 把高依赖或高风险工具隔离到独立进程
