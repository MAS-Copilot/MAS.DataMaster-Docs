.NET事件和委托：

  • 直接性：事件是直接定义和触发的，订阅者必须直接知道发布者，形成了比较紧密的耦合
  • 简洁性：对于简单的场景，使用标准的.NET事件更直接、简单
  • 组件耦合：发布者和订阅者之间的耦合度较高，因为订阅者需要具体知道是哪个类或实例发布的事件

事件聚合器（Event Aggregator）：

  • 解耦合：发布者和订阅者之间不直接通信，而是通过事件聚合器，这降低了组件之间的耦合度
  • 灵活性：订阅者可以在运行时选择订阅或取消订阅事件，而不需要修改发布者的代码
  • 管理性：事件聚合器作为一个中心点管理所有的事件和订阅，更容易进行事件的管理和调试

IEventAggregator

行为

- Step 1: 在 MAS.SharedLibrary 中定义事件（跨模块、避免循环依赖），便于集中管理以及访问

public class DataUpdatedEvent{
    public string UpdatedData { get; set; }
}

- Step 2: 订阅者可以在任意位置订阅事件

public class MainViewModel{
    private readonly IEventAggregatorService _eventAggregator;

    public MainViewModel(IEventAggregatorService eventAggregator){
        _eventAggregator = eventAggregator;
        _eventAggregator.Subscribe<DataUpdatedEvent>(OnDataUpdated);
    }

    private void OnDataUpdated(DataUpdatedEvent e){
        Console.WriteLine($"数据已更新: {e.UpdatedData}");
    }
}

- Step 3: 发布者可以在任意位置发布事件

public class DataService{
    private readonly IEventAggregatorService _eventAggregator;

    public DataService(IEventAggregatorService eventAggregator){
        _eventAggregator = eventAggregator;
    }

    public void UpdateData(){
        // 执行数据更新操作
        // ...

        _eventAggregator.Publish(new DataUpdatedEvent { UpdatedData = "新的数据内容" });
    }
}

信息

使用已存在的 DataUpdateEvent<T>(T data) 泛型事件类足以满足多种数据类型的需求

public class DataUpdateEvent<T>(T data) {
    public T Data { get; } = data;
}

消息队列

自3.9.0及以上版本中引入的消息队列模式

目的

  • 解决标准Pub/Sub的瞬时消息丢失问题：允许延迟订阅者（或在订阅后）通过持久化队列获取历史消息
  • 支持解耦：发布者和消费者通过约定（文档/业务键）知晓publisherId和messageId，避免模块耦合
  • 隔离性：消息按publisherId隔离，仅匹配ID的订阅者/消费者可见
  • 实时+手动：发布持久化消息时，既存入队列（防丢失），也实时通知匹配订阅者（无需轮询）。消费者可手动GetMessage获取遗漏消息
  • 过期管理：默认30分钟过期，自动移除，防止内存无限增长（理论支持上万条消息）
  • 线程安全：使用线程安全字典ConcurrentDictionary和lock，确保多线程环境稳定
  • 存储：当前内存实现（高效），未来可扩展到数据库（Redis）以支持持久化重启

行为

- Step 1: 在 MAS.SharedLibrary 中定义事件（跨模块、避免循环依赖），便于集中管理以及访问

public class DataUpdatedEvent{
    public string UpdatedData { get; set; }
}

Step 2: 订阅特定消息（实时通知）

public class ConsumerViewModel {
    private readonly IEventAggregatorService _eventAggregator;

    public ConsumerViewModel(IEventAggregatorService eventAggregator) {
        _eventAggregator = eventAggregator;
        // 订阅特定publisherId和messageId（通过文档约定知晓）
        _eventAggregator.SubscribeSpecific<DataUpdatedEvent>(OnSpecificDataUpdated, "PublisherModule1", "UpdateMessage123");
    }

    private void OnSpecificDataUpdated(DataUpdatedEvent e) {
        Console.WriteLine($"特定消息更新: {e.UpdatedData}");
    }
}

Step 3: 发布持久化消息

public class PublisherService {
    private readonly IEventAggregatorService _eventAggregator;

    public PublisherService(IEventAggregatorService eventAggregator) {
        _eventAggregator = eventAggregator;
    }

    public void PublishUpdate() {
        var eventData = new DataUpdatedEvent { UpdatedData = "持久化数据" };
        // 使用约定ID发布，存队列 + 实时通知匹配订阅者，默认过期30分钟
        _eventAggregator.PublishPersistent(eventData, "PublisherModule1", "UpdateMessage123");
    }
}

Step 4: 手动获取消息（如果订阅延迟或需重试）

public class LateConsumer {
    private readonly IEventAggregatorService _eventAggregator;

    public LateConsumer(IEventAggregatorService eventAggregator) {
        _eventAggregator = eventAggregator;
    }

    public void FetchMissedUpdate() {
        var missedEvent = _eventAggregator.GetMessage<DataUpdatedEvent>("UpdateMessage123", "PublisherModule1");
        if (missedEvent != null) {
            Console.WriteLine($"手动获取: {missedEvent.UpdatedData}");
        }
    }
}

注意事项

  • publisherId和messageId为string，便于业务键
  • 唯一性：发布者需确保messageId在publisherId下唯一（覆盖旧消息）
  • 过期：expiration为null时默认30分钟
  • 性能：内存存储支持上万条；若需清理，可监控字典大小
  • 解耦：ID通过文档/配置约定，避免直接通信
